## 大まかな構想

人間と複数の AI が協調して開発を進める **自律型並行開発基盤**。人から指示・承認を行い、AI 群がタスクの分解・実行・相互レビュー・安全審査を分担する。

- **2 台構成**: 司令塔の **Mac mini**（判断・制御）と、実行機の **MacBook Pro**（重量・軽量タスクの実行）。今後、1台での廉価版を検討中。
- **通信は SSH のみ**: マシン間はポート開放・REST API を用いず SSH。外部との通信は Slack Private channel に限定。
- **モデルは役割で選ぶ**: 「軽量（light）／重量（heavy）」の 2 区分でルーティングし、各役割に最適なモデルを割り当てる（モデルは差し替え可能）。

## マシン構成

| 役割 | マシン | スペック | 主な担当 |
|------|--------|---------|---------|
| 司令塔 (Command Center) | Mac mini M4 Pro | 64GB / 2TB | sa-ru / ya-ta / u-zu |
| 実行機 (Execution Hub) | MacBook Pro M4 Max | 128GB / 8TB | worker LLM 群 / qu-e |
| 人間インターフェース | Private Slack App | Socket Mode | 指示・承認 |
| マシン間接続 | 10GbE 直結 + Tailscale SSH | 在宅=10GbE 直結 / 外出=VPN、自動切替 | 双方向 SSH |

## コンポーネントと役割

| コンポーネント | 役割名 (src) | 配置 | 役割概要 |
|--------------|------------|------|---------|
| **sa-ru** | orchestrator | Mac mini | 司令塔。タスク受領 → 分解依頼 → worker 振り分け。PTY 制御・プロセス管理（ログは各コンポーネント個別出力、中央集約器なし） |
| **ya-ta** | ai_gateway | Mac mini | sa-ru が import するライブラリ。タスクの分解・分類・リスク判定・モデルルーティング |
| **qu-e** | sentinel | MBP | 守護プロセス。Tier 2 コード審査・ヘルスチェック・ファイル監査 |
| **u-zu** | slack_bot | Mac mini | 唯一の外部通信。人間からの指示受付・承認通知（Socket Mode） |
| **approval-pipeline** | sa-ru 配下 | Mac mini → MBP | y/n 三段階リスク承認。sa-ru から呼び出される |

> 命名規約: デプロイ先（pyinfra）は**コンポーネント名**（古事記由来）、ソース（src）は**役割名**。

## アンインストール方法と仕組み

本システムは pyinfra による冪等デプロイで構築される（[01](01-common-base.md)〜[08](08-approval-pipeline.md)）。構築の各ステップ（**pyinfra の自動操作・ユーザーの手動操作の両方**）は、完了ごとに **インストール・マニフェスト** へ構造的に記録される。アンインストールはこのマニフェストを **逆順（LIFO）で再生** して撤去する。**常駐サービスの停止を最優先** に行う（launchd の `KeepAlive` で自動再起動するため、プロセスを kill するだけでは復活する）。

> **記録と逆順撤去の仕組み**（マニフェストのスキーマ・保存先・記録タイミング・手動ステップの会話完了確認フロー）は、インストール来歴タスクで設計する。仕組みが整うまでは、下記「痕跡一覧」を根拠に手動で逆順撤去する。

### マニフェストに記録される主な痕跡

| 種別 | 配置 | 備考 |
|------|------|------|
| 常駐サービス (launchd) | `~/Library/LaunchAgents/com.taka-ma.{sa-ru,u-zu,qu-e}.plist` | sa-ru・u-zu=Mac mini / qu-e=MBP。ya-ta は sa-ru 内ライブラリのため非サービス |
| アプリ本体 | `/opt/taka-ma/`（sa-ru / ya-ta / u-zu / qu-e / config / logs / data / models） | `data/` にタスク・承認履歴 |
| Python 仮想環境 | `/opt/taka-ma-env/` | uv で作成 |
| ローカル LLM (ollama) | [初期 LLM の配置と用途](#初期-llm-の配置と用途) の各 ollama モデル | 容量大。共有 ollama に注意 |
| SSH 設定 | `~/.ssh/{taka-ma-cluster,taka-ma-cluster.pub,config}`、`/etc/ssh/sshd_config.d/taka-ma-cluster.conf`、`sshd_config` の Include 行 | 双方向 |
| 静的 IP | `networksetup` で設定した 10GbE 直結 IP | 直結を使う構成のみ |
| Homebrew / 外部資産 | ollama・node・gh 等の汎用ツール、Slack App、Claude / Gemini API キー、Tailscale | 共有・外部のため自動削除しない |

### 逆順撤去の順序

撤去はマニフェストの逆再生（LIFO）で行う。具体的な実行コマンドはアンインストール runner（`pyinfra/lib/uninstall.py`）がマニフェストから生成・実行する。手動で行う場合も以下の順序を守る。

1. **常駐サービスの停止・解除（最優先）** — launchd サービス（Mac mini=sa-ru・u-zu / MBP=qu-e）を停止・解除する。`KeepAlive` による再起動を止めるため最初に行う。
2. **データ退避** — 残したい場合のみ `/opt/taka-ma/data/`（タスク・承認履歴）をバックアップ。
3. **アプリ・仮想環境の削除** — `/opt/taka-ma`・`/opt/taka-ma-env` を削除。
4. **ローカル LLM の削除** — 両マシンの ollama モデル（[初期 LLM の配置と用途](#初期-llm-の配置と用途) 参照）を削除。本システム専用に ollama を入れた場合は ollama 自体も停止。
5. **SSH 設定の復旧** — クラスタ用 sshd 設定・鍵・`~/.ssh/config` を撤去し sshd を再起動。**遠隔作業中は自分の接続を切らないよう最後に実施する。**
6. **静的 IP の解除** — 10GbE 直結を使わない場合は DHCP へ戻す。

### 共有資源・外部資産（自動削除しない）

- **Homebrew パッケージ**（`node` / `gh` / `git` / `python@3.12` / `tmux` 等）は汎用ツール。他用途がないと確認できた場合のみ個別に `brew uninstall`。
- **Slack App** は Slack 管理画面でアンインストール（トークン失効）。
- **Claude / Gemini の API キー** は各プロバイダ側で失効させる。
- **Tailscale** は他用途がなければアプリ側でログアウト・削除。

## リリースノート（予定）

初回公開以降に予定している更新の一覧と投入順。本システムは各自の PC 上に構築するもので、初回公開後の改善は Update（修正）として反映する。優先度と依存関係で順序を決めている。着手はいずれも初回リリース後。

### 投入順とマイルストーン

| 順 | マイルストーン | 更新項目 | 位置づけ |
|----|--------------|---------|---------|
| 1 | 信頼性の土台 | sa-ru の待受キュー（FileQueue）堅牢化 — 未知 status の滞留・done 無しローテートへの対策 | 機能追加の前に実行基盤を固める |
| 2 | ルーティング品質 | LLM 層のモデル抽象化 — run_ollama の `--hidethinking` 常時付与を是正（非思考モデルでの無害性を実機検証）し、思考／JSON 等のモデル固有オプションを ya-ta.yaml のモデル別設定へ切り出す | 判定・切替の下地。モデル差をコードから設定へ追い出す |
| 3 | ルーティング品質 | タスク判定ログ Phase 2 — 分類プロンプトの自動改善バッチ | 既存経路の判定精度を上げる |
| 4 | ルーティング品質 | Cloud AI 選択時のモデル切替判定とプロバイダのツール活用 — ya-ta の判定内で Cloud 版（例 Claude）のモデル階層（超 Heavy / Heavy / Middle）を選び、プロバイダのサービス/ツールを役割別（コード開発／UI 開発 等）に使い分ける | 外部 AI を担当タスクに応じて最適に振る |
| 5 | ルーティング品質 | 難易度に応じた heavy モデルの自動段階選択 — タスク難易度から heavy 内のモデル階層（通常 heavy／最難関）を ya-ta が自動選択。現状は `:モデル名` の明示指定のみ | 前項の延長。明示指定を自動判定へ引き上げる |
| 6 | 会話継続性 | intent 連続捕捉機構 — 会話ストリームを一次ソースにし、ドリフトを AI 検出 → 人が承認 → 追記 | 次項の下地 |
| 7 | 会話継続性 | 依頼（会話）とタスクの継続紐づけ — conversation_id / parent_task_id の永続化、会話セッションの永続化 | 「設計 →（時間をおいて）実装」を同一依頼として継続できるようにする |
| 8 | チャット抽象化 | ChatPort 化 — Slack 依存を撤去し任意のチャットに対応 | 会話体験が固まってから外部インターフェースを差し替え可能に |
| 9 | マルチモーダル生成 | 動画・音楽などの生成基盤を外部サービス登録制で追加 — 生成サービス/モデルを登録して利用（解析＝入力に続く、生成＝出力側の拡張） | 新機能カテゴリ。マルチモーダル解析の次段 |
| 10 | スケール検証 | 複数ワークスペース運用（team_id 別トークン）の検証 | 機能が出そろってから多重運用を実証 |
| 11 | スケール検証 | Mac mini ↔ MacBook Pro の 10Gbps 直結による構築・実証 | ハードウェア最適化。独立項目 |

### 検討中（投入順の外）

- **CI/CD パイプライン**: 本システムは各自の PC に構築するものでサーバー展開を伴わない。CI/CD が有効に機能する構成か（自動テスト・デプロイの適用範囲）を含め、採否から検討する。
