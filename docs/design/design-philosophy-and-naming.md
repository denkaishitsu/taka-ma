# 設計思想と命名体系

**作成日**: 2026-04-27
**位置付け**: 本プロジェクトの根本思想と、命名体系の決定根拠を一元化した参照資料。
設計書・構築手順書・ソースコードは本ドキュメントに整合させる。

## 1. 議論の発端

設計書および構築手順書04のレビューを進める中で、ユーザーから以下の疑問が提起された:

> 「OpenClaw を採用する方向でいるが、何が良くて世の中で騒がれたのか？」

この問いをきっかけに、本プロジェクトにおける「OpenClaw」の位置付け、そして本システム全体の設計思想そのものを再点検する議論に発展した。

---

## 2. アシスタント側の誤認と修正経緯（透明性のため記録）

議論の前段で、アシスタント（Claude）は以下の誤認を犯した。記録として残す:

1. **第1の誤認**: ローカルのソース構造（`src/openclaw/`）のみを根拠に「OpenClaw は自作の内製名であり、世間で騒がれている OSS は存在しない」と断定
2. **第2の誤認**: ユーザーから訂正を受けても、Web上の一次ソースを確認せず2回押し返した
3. **修正契機**: ユーザーが `https://openclaw.ai/` を提示したことで、実在 OSS であることを確認

実在 OSS の OpenClaw（Peter Steinberger 作、2025年11月公開）は、GitHub Star 247,000+ を約60日で獲得した、極めて急成長中のパーソナルAIアシスタント基盤である。

**教訓**: 固有名詞・新製品については、自身の知識を信じず一次ソースを当たること。本リポジトリの開発方針「AI 出力を鵜呑みにせず、必ず一次ソースで検証する」は、アシスタント自身にも適用される。

## 3. 設計思想のコア

ユーザーから明示された、本プロジェクトのコア哲学:

> **「人とAIの協調」**

### 3.1 「協調」の成立条件

「協調」は以下の状態を指し、いずれも欠けると成立しない:

- **AIに裁量がある**: 些末な判断まで人に上げない
- **人は重要な分岐点でのみ承認に介入する**: 全件承認ではない
- **責任は最終的に人が引き受ける**: AIに丸投げしない
- **人は監督を絶対にやめない**: 放任もしない

「全権委譲（放任）」でも「全件承認（過干渉）」でもない、**第三の状態**こそが「協調」である。

### 3.2 AI Gateway がコアである必然性

このフィロソフィーを物理的に成立させる装置が AI Gateway（後述 ya-ta）である:

- **裁量と承認の境界線を、タスクごとに動的に決める装置**
- リスク低 → AI が自走、人は事後通知のみ
- リスク高 → 人に Slack で承認を仰ぐ

AI Gateway がなければ「全件承認（過干渉）」か「全件自走（無責任）」の二択しかない。
**AI Gateway は「協調の境界線」を物理的に成立させる装置であり、本プロジェクトで OSS による代替が効かない本質的独自価値である。**

## 4. 汎用協調基盤としての位置付け

本システムは「Mac mini + MBP の開発環境」ではなく、**「人とAIの協調の汎用基盤」のリファレンス実装**である。
IT開発はその最初の応用に過ぎない。

### 4.1 汎用性の成立条件

以下の3条件を満たす任意の領域に、同じ骨格構造で適用可能:

- タスクにリスク勾配がある
- 人間が最終責任者である必要がある
- 複数の専門能力（AIまたはツール）に振り分けたい

### 4.2 想定応用領域

| 領域 | Slack(指示) | Gateway(分類) | Worker(実行) | 査読 | 承認ゲート例 |
|---|---|---|---|---|---|
| **IT開発** | "ログインフォーム作って" | コーディング/重 | Claude Code | 査読AI | 本番デプロイ・DB変更 |
| **ロボティクス** | "倉庫B区画の棚卸し" | 動作種別/危険度 | ロボットアームAPI / AMR制御 | センサーログ整合性 | 人間近接動作・高出力 |
| **コンサル** | "X社の市場分析レポート作って" | 機密度/外部公開 | Deep Research / Claude / 社内DB | 事実関係査読 | クライアント提出物 |
| **為替約定** | "USDJPYを150で逆指値" | 通貨ペア/サイズ | 取引所API（複数）| ポジション整合性 | サイズ閾値超 |

**変わるのは Worker レイヤと Gateway の分類タクソノミのみ**。骨格は完全に再利用される。

### 4.3 領域固有に必要となる追加実装

楽観しすぎないため、領域ごとに必要な追加層を明記:

1. **リスク分類タクソノミ**: 領域別のリスク基準
2. **レイテンシ要件**: 為替=ms、ロボティクス=ms〜秒、コンサル=時間
3. **規制境界**: 金商法・薬機法等
4. **物理安全装置**: ロボティクス等は物理ガード必須
5. **監査ログ**: 法定記録要件

これらは基盤の上に乗せる「領域アダプタ」として設計可能で、骨格自体は変えなくてよい。

## 5. OpenClaw（OSS）採用是非の結論

### 5.1 結論

**OpenClaw（OSS）は採用しない。** 理由は以下:

1. **思想の不一致**: OSS OpenClaw は「人がAIに権限を委譲して便利になる」思想（50チャンネル・永続記憶・自己拡張skill・sandbox緩和）。本プロジェクトの「人が権限を絶対に手放さない」思想と方向が逆
2. **AI Gateway との二重化**: OSS OpenClaw 自身がルーターであり、AI Gateway と機能が競合する。どの統合パターンを採っても思想か機能のいずれかが壊れる
3. **過剰特化**: OSS OpenClaw は「個人AIアシスタント」というドメインに最適化されており、汎用基盤の骨格に据えるとドメイン特化機能が常に「使われない重荷」になる
4. **沈没コストの観点**: 設計と初期実装は既に完了しており、自作継続コストは比較対象として小さい

### 5.2 補足

OSS OpenClaw の機能群を Worker 側（IT開発専用ワーカーの一つ）として組み込むのは別判断として残る。
骨格に据えないという結論である。

## 6. 命名体系の決定

「OpenClaw」という名称は、OSS OpenClaw との混同を避けるため廃止。
古事記の神格・概念から取り、英語風に表記する命名体系を採用する。

### 6.1 命名規約

- **元名から「意味の核（修飾語・形容語）」を取り、本体名詞・敬称接尾辞を落とす**
- **2モーラ単位でハイフン連結**
- **英語話者が読みやすい綴り**（厳密なヘボン式に縛られない。例: kue → qu-e）

### 6.2 確定した命名

| 役割 | 新名 | 旧名 | 古事記の元 | 残した部分 | 落とした部分 |
|---|---|---|---|---|---|
| リファレンス実装（リポジトリ・製品名） | **taka-ma** | taka-ma | 高天**原** | 高天 | 原（場所接尾辞） |
| オーケストレーター | **sa-ru** | o-moi（旧 OpenClaw） | **猿**田彦 | 猿 | 田彦（地名・敬称接尾辞） |
| ゲート（リスク分類・ルーティング判定） | **ya-ta** | AI Gateway | 八咫**烏** | 八咫 | 烏（本体名詞） |
| 査読 | **qu-e** | Sentinel | 久延**毘古** | 久延 | 毘古（敬称接尾辞） |
| Slack インターフェース | **u-zu** | slack-bot | 天**宇受**売命 | 宇受 | 天（接頭辞）・売命（本体名詞・敬称接尾辞） |

### 6.3 各名称の象徴的意味

#### taka-ma（高天原）
神々が集議し、地上を監督する天の平原。
**「神々（AI群）が集い、人が監督する場」** を体現する基盤名。

#### sa-ru（猿田彦命）
天孫降臨の際、天の八衢（やちまた＝分かれ道）に先回りして待ち受け、ニニギ一行を地上の高千穂まで先導した国津神。道祖神・道の神の祖。
**「常駐してイベントを見張り、起きたタスクを適切な実行者へ振り分ける道案内（司令塔）」** という現アーキテクチャの中核挙動と意味が一致する。

> 旧名は o-moi（思金神＝判断の取りまとめ）。オーケストレーターの本質は「思考・取りまとめ」より「常駐し道を指し示す案内」だと再定義し、sa-ru（猿田彦）へ改名した。

#### ya-ta（八咫烏）
神武天皇を導いた三本足の鴉。難所を通す案内者。
**「道を開き、最適な経路に導く」** Gateway の役割を直接体現。

#### qu-e（久延毘古）
「久延」は古語「くゆ／くえる」由来で**朽ちる・崩れる**の意。
案山子の風化した姿を指す。久延毘古は古事記原文で**「天下の事を悉に知れる神」**と記される、動かないが全てを見抜く神。
**「実行はせず、全コードを読み判断する査読AI」** の役割と1対1対応。

#### u-zu（天宇受売命）
天岩戸に隠れた天照大神を誘い出すため、岩戸の前で舞い、八百万の神々の哄笑を引き起こした芸能・神楽の女神。神と人（神々の世界と外界）をつなぐ最前面の演じ手。
**「人と AI 群（taka-ma）の接点に立ち、人の指示を受け取り状況を返す Slack インターフェース」** の役割と一致する。

> 旧名は slack-bot（チャネル実装名）。命名規約に従い天宇受売命由来の u-zu へ改名した。

### 6.4 査読役を「qu-e」（単独）とした根拠

候補としては「qu-e」と「qu-e-biko」があったが、命名規約「**修飾語のみ残し、本体名詞・敬称を落とす**」に厳密に従い **qu-e（久延）単独** に確定。

- ya-ta が本体名詞「烏」すら落としている前例があるため、qu-e が敬称「毘古」を落とさない理由はない
- qu-e-biko を維持すると、qu-e のみ完全名となり規約整合性が崩れる
- 同音の懸念（食え／悔い）は、英語表記運用と文脈で吸収可能

### 6.5 体系の意味論的整合

taka-ma / sa-ru / ya-ta / qu-e / u-zu は、いずれも**記紀神話（高天原を中心とする神話圏）**の神格・概念であり、神話圏として閉じている。
意味論が単一神話圏に統一されることで、命名体系全体が一貫した世界観を持つ。

### 6.6 表記規則: ハイフンとアンダースコアの使い分け

命名体系の表記は **`taka-ma` / `sa-ru` / `ya-ta` / `qu-e`**（ハイフン区切り）を原則とする。
ただし Python の構文制約（識別子にハイフンを許容しない）により、すべての場面で同じ表記は使えない。
以下のルールで用途別に使い分ける。

#### 用途別表記

| 用途 | 表記 | 例 |
|---|---|---|
| ヒトが読む場面（リポ名・YAMLキー・ファイル名・ドキュメント・ログ表示） | **ハイフン** | `taka-ma`, `ya-ta.yaml`, `config["ya-ta"]` |
| Python が読む場面（パッケージ名・モジュール名・変数名・import 文） | **アンダースコア** | `import ya_ta`, `src/ya_ta/` |

#### 根拠

- Python の識別子はハイフンを許容しない（マイナス演算子と衝突）
- 外部識別子（パッケージ名・URL等）はハイフンが世間の主流（例: PyPI 名 `scikit-learn` / import 名 `sklearn`）
- 命名体系のハイフン表記を維持しつつ Python の制約に従うには、用途別使い分けが現実解

#### 適用例

| 対象 | 表記 |
|---|---|
| 設定ファイル名 | `ya-ta.yaml`, `sa-ru.yaml`, `qu-e.yaml` |
| YAML キー | `ya-ta:`, `sa-ru:`, `qu-e:` |
| Python ディレクトリ・パッケージ名 | `src/ya_ta/`, `src/sa_ru/`, `src/qu_e/` |
| Python 識別子（変数・関数） | `ya_ta`, `sa_ru`, `qu_e` |
| Python クラス名 | `YaTa*`, `SaRu*`, `QuE*`（CamelCase 化） |
| ログ・jsonl ファイル名 | `ya-ta-decisions-{date}.jsonl` |
| デプロイ先パス | `/opt/taka-ma/ya_ta/`（import 想定のためアンダースコア） |

#### 概念説明文での扱い

「AI Gateway」「ゲート」「オーケストレーター」等の概念用語は、新名と併記する形で**初出に限り旧名・概念名を示し、以降は新名で記述する**。

例:
> ya-ta（AI Gateway）はタスクの分解・分類・リスク判定を行う。

これにより読み手の理解可能性を維持しつつ、システム内識別子は新名で統一できる。

## 7. アシスタント運用への含意

本フィロソフィーは、ユーザーがアシスタント（Claude）に求める振る舞いにも適用される:

- **裁量範囲内では自分で判断・実行する**: 些末な確認をやめる
- **見立てを持ち、押されても証拠が出るまで変えない**: 忖度による翻意は禁止
- **判断が外れたら認め、なぜ外れたかを言語化する**
- **重要な分岐でのみ承認を仰ぐ**

「全件承認（過干渉）」のアシスタントは、本プロジェクトの目指す協調から最も遠い存在である。
