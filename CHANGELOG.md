## [0.3.0] - 2026-07-23

### Added

- 実行前に「実行計画」を提示して承認を取るゲートを追加。段組み（どの作業を並行し、どこで待ち合わせるか）と各作業の担当モデル・深さが着手前に分かる。承認した計画はそのまま実行され、提示内容と実際に走る内容がずれない
- 提示された計画を、承認前にチャットで訂正できる。簡易な指定（担当モデルや深さの変更）は即座に反映して計画全体を再提示し、自然な文章での指示は変更点だけを差分で確認できる。訂正はスレッド内・スレッド外のどちらの投稿でも受け付ける

### Changed

- 依頼の振り分けを「軽い/重い」の2値から、実行形態（その場で答える／エージェントとして作業する）と深さ（浅い／深い）の2軸に再設計。判定はどのモデルを使うかを直接決めず、設定ファイルの対応表が担当モデルを決めるため、モデル構成の変更が設定編集だけで済む
- 実行が失敗・タイムアウトした場合、より能力の高いモデルへ段階的に引き上げて再試行するようになった。作業側からの「手に負えない」という申告も引き上げの引き金になる
- その場で答えるだけの依頼と、作業を伴う依頼を別レーンで処理するよう変更。片方の混雑がもう片方を待たせない

### Documentation

- 設計書を上記の振り分け2軸化・計画確認ゲートに合わせて改訂
- 構築手順書（AI Gateway）を振り分け2軸の設定手順へ更新
- 構築手順書（Sentinel）へ、終了済み作業ディレクトリの自動整理（削除対象と除外条件・確認手段・起動時と日次の実行タイミング）を追記
- システム俯瞰へ「リリースノート（予定）」として今後の投入順とマイルストーンを追加

## [0.2.1] - 2026-07-19

### Added

- タスクごとの作業ディレクトリ（workspace）を、そのタスクが完了したら自動で退避・整理する仕組みを追加。使い終わった作業領域が溜まり続けない。対象や保持数は設定ファイル（yaml）で指定する
- チャット受信（常時接続）の死活監視を追加。接続が無応答になっても内部で検知して復帰でき、「メッセージを送ったのに反応しない」状態が続くのを防ぐ

## [0.2.0] - 2026-07-17

### Added

- コミット前に変更内容を自動監査し、問題を検出したらコミットを止めるゲートを追加。監視対象のリポジトリを設定で指定でき、実際の開発リポジトリの変更を動的に監視する

### Changed

- 運用パラメータの既定値をコード内から排除し、設定ファイル（yaml）を単一の正本に統一。挙動を変えたいときはコードではなく設定を編集する
- 会話・タスク分解のローカル LLM 呼び出しを `ollama run` の都度起動から HTTP API（/api/generate）へ変更。モデル常駐（keep_alive）と KV キャッシュ再利用により、会話が長くなるほど毎ターン遅くなる問題を解消
- タスク完了通知の切り詰め（先頭 500 文字のみ）を廃止し、実行結果の全文を受け取れるように変更
- 実行結果を会話文脈へ還流するよう変更。タスク完了後に「さっきの結果はどこ」等の後続質問へ答えられる
- 会話履歴をディスクへ永続化。プロセス再起動や時間経過で会話の文脈が失われない

### Fixed

- 着手確認が一定時間で自動タイムアウトし「もう一度指示してください」を強いる挙動を撤廃。人間がボタンで決着するまで待ち続ける
- LLM 呼び出し失敗が一律「内部エラー」としか通知されなかった問題を修正。タイムアウトと接続不可を区別して原因を通知する
